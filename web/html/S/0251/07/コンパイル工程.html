コンパイル工程

ソースコードをコンパイルして実行可能プログラムを生成するために、コンパイラは以下の5つの処理を行います。
  1. 字句解析
  2. 構文解析
  3. 意味解析
  4. 最適化
  5. コード生成

字句解析

英語では「lexer」と言います。
コードを最小単位のパーツに区切る作業を言います。
最小単位のパーツをトークンと呼び、C#では「var」「=」「(」「10」「+」「25」「)」「;」などが該当します。
この段階では、トークンを生成することを目的としています。

構文解析

英語では「parser」と言います。
字句解析で生成したトークンに階層構造に整えます。

かなりややこしい部分ですが、最小単位であるトークンと、トークンに階層構造をもたせた節の部分をノード、または構文ノードと呼びます。

例えば、「var a = 1 + 2;」という文では、「var」「a」「=」「1」「+」「2」「;」と、「var a」「1 + 1」「var a = 1 + 1;」がノードとして扱われます。
各ノードは、「Kind」と呼ばれるノードが該当する性質を表すプロパティを有します。

また、構文のことを「Syntax」、構文ノードのことを「SyntaxNode」、構文階層のことを「SyntaxTree」と英語で言います。

意味解析

英語では「semantic analyze」を言います。
構文解析で生成した構文階層をもとに、実際の命令を行う対象を整理します。
この段階では意味を持つシンボル(Symbol)を生成します。
シンボルには実際にコードを実行する際に使用する情報が含まれているため、「定義へ移動」や「全ての参照を検索」のAPIが使えます。

最適化

コードの冗長部分を削って、コードのパフォーマンスを高める処理です。

コード生成

実際に実行可能プログラムを生成します。

ソリューション・プロジェクト・ドキュメント

ソリューションは単一、ないしは複数のプロジェクトを有し、プロジェクトは単一の、ないしは複数のドキュメントを有する階層構造をとっています。

画像はgithubより。

ワークスペースについては後ほど説明します。

ソリューション

プロジェクトのコンテナとしての役割を果たします。
「*.sln」ファイルによって、当該ソリューションが含むプロジェクトを指定します。

プロジェクト

コードのコンパイル単位です。
「*.csproj」「*.vbproj」ファイルによって、当該プロジェクトが使用するファイルを指定します。
このファイルに記載された内容によってコンパイル時に必要なファイルや、ライブラリを特定します。

ドキュメント

プロジェクトに含まれるファイルひとつひとつが該当します。
